Proposal for review

Live GPS tracking for field technicians

A lightweight way for dispatch to see, on a live Google Map, exactly where each technician is while a roadside job is running — that keeps working through mobile dead zones and records the distance driven per call.

01 — Problem statement

Once a technician leaves, we’re blind

Our technicians drive out to roadside breakdowns to change and repair tyres — for example service event 07885993, a trailer tyre job off I-95 in New Jersey. From the moment a technician leaves until they phone in, dispatch has no idea where they actually are. That costs us in concrete ways:

  • No live position or ETA — we can’t tell a waiting customer when help will arrive.
  • Can’t dispatch the nearest technician — when a new call comes in, we’re guessing who’s closest.
  • No confirmation of arrival or on-site time — we rely on the technician to call it in.
  • No distance record per job — the miles driven for a call aren’t captured automatically.

We need a live map of every technician that keeps recording through poor-signal areas and measures distance for each job.

02 — Solution

Track from the technician’s phone, show it on a map

Add lightweight location tracking to the web app technicians already use — nothing to install, no devices to buy. The phone’s GPS reports to our own backend, which feeds a live Google Map for dispatch. Four steps:

1

Capture

The technician’s existing web app reads GPS in the browser — no native app, no hardware to fit.

2

Send

Readings post to our backend every ~10–15s. If offline, they queue on the phone and send on reconnect.

3

Store

A Python backend validates and stores each point, and keeps a running distance for the job.

4

Display

Dispatch sees every technician as a live marker on Google Maps, updating in real time.

Why dead zones don’t break it

GPS works without a signal — the phone keeps recording even with no internet. Readings are held on the device and sent the moment the connection returns, so the trail fills in with the real positions driven. Nothing is lost during an outage.

Distance is measured from the recorded track itself, so it’s accurate to what the vehicle actually drove and adds no extra cost.

03 — Recommendation

What we should build

Build the browser-based tracker now, hosted on Google Cloud

It delivers live tracking quickly using the app technicians already have, at low cost. We considered two alternatives and don’t need either today: a native mobile app would only be worth building later if we need tracking to continue with the phone locked in a pocket; dedicated in-vehicle GPS hardware would track the truck rather than the person and carries a per-vehicle cost.

Supporting decisions we’ve made to keep it simple and cheap: measure total distance driven per job; keep everything in one cloud (Google) for a single bill; push live updates with a lightweight server stream; and skip any extra caching layer in version 1.

Build the browser tracker on Google Cloud now; revisit a native app only if we ever need tracking while the phone is locked.

Why the cost stays low

Google Maps is effectively free here because only dispatchers load a map, and moving the markers around doesn’t add to the bill — technicians’ phones don’t show a map at all, they just send location. We also calculate distance ourselves, so there’s no paid mapping call for it. (Figures are representative and shift a little with region and usage; Google adjusts its pricing periodically.)

To proceed

What a green light needs

  • A Google Maps API key with billing enabled, locked to our domain (with daily caps and a budget alert).
  • An HTTPS web address for the technician page — provided automatically by Cloud Run.
  • Sign-off on tracking technicians (HR / legal), including whether it’s on-shift only.
  • Then engineering builds the database, the backend, and the small in-browser tracker — which drops into our existing app regardless of how it’s built (about 150 lines of standard code), so nothing has to be rewritten.
